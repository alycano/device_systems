Descripción de la actividad
Cada aprendiz deberá evolucionar la aplicación device_systems, desarrollada en la actividad
anterior, incorporando mecanismos de seguridad y protección propios de una API REST
profesional.
En la versión anterior, device_systems integró FastAPI, SQLAlchemy, Alembic, modelos
relacionados, migraciones, consultas con joins y filtros. En esta nueva actividad, el aprendiz deberá
fortalecer la API agregando:
• Validaciones avanzadas con Pydantic v2.
• Middleware personalizado.
• Configuración de CORS.
• Hash seguro de contraseñas con passlib.
• Autenticación con OAuth2 y JWT.
• Protección de rutas mediante dependencias.
• Rate limiting para limitar peticiones abusivas.
• Mejora de la seguridad general de la API.
La aplicación seguirá llamándose:
device_systems
Los recursos principales continuarán siendo:

GFPI-F-135 V04

/users
/devices
/loans
Y se agregará el recurso de autenticación:
/auth
El propósito de esta actividad es que el aprendiz transforme la API en una versión protegida,
controlada y preparada para interactuar con clientes frontend de forma segura.

Reto Integrador – Seguridad y Autenticación en device_systems
Evolucionar la API REST device_systems para implementar autenticación, autorización básica,
protección de rutas, validaciones robustas, configuración CORS, middleware personalizado y rate
limiting.
La API debe permitir:
• Registrar usuarios con contraseña segura.
• Almacenar contraseñas usando hash.
• Autenticar usuarios mediante login.
• Generar tokens JWT.
• Proteger rutas privadas con OAuth2.
• Restringir operaciones según rol.
• Configurar CORS para permitir clientes autorizados.
• Aplicar middleware global para trazabilidad y cabeceras.
• Limitar el número de peticiones por cliente.
• Mejorar los schemas con Pydantic v2.

Fase 4 – Mejorar el modelo User
Actualizar el modelo User para soportar autenticación.
Campos mínimos adicionales:
Campo Tipo Restricción
hashed_password String Obligatorio
role String Obligatorio
is_active Boolean Valor por defecto True
Roles sugeridos:
admin
support
user
El campo hashed_password nunca debe exponerse en los response models.
Después de modificar el modelo, el aprendiz debe generar una migración con Alembic:
alembic revision --autogenerate -m "add authentication fields to users"
alembic upgrade head

Fase 7 – Implementar autenticación OAuth2 con JWT
Crear endpoints en:
app/auth/auth_routes.py
Endpoints obligatorios:
POST /auth/register
POST /auth/login
GET /auth/me

GFPI-F-135 V04

POST /auth/register
Debe permitir crear un usuario con contraseña segura.
Debe validar:
• Nombre obligatorio.
• Email válido.
• Email único.
• Contraseña segura.
• Rol permitido.
Debe guardar:
• Nombre.
• Email.
• Hash de contraseña.
• Rol.
• Estado activo.
No debe guardar la contraseña en texto plano.
POST /auth/login
Debe permitir autenticar usuario y retornar token JWT.
Respuesta esperada:
{
"access_token": "token_generado",
"token_type": "bearer"
}
GET /auth/me
Debe retornar los datos del usuario autenticado, usando el token enviado en:
Authorization: Bearer <token>
No debe retornar hashed_password.
